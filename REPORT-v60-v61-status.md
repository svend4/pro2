## 1. Краткая сводка проекта

YiJing-Transformer — трансформерная архитектура с индуктивным смещением на основе геометрии гиперкуба Q6 = {-1,+1}^6. 64 гексаграммы И-Цзин = вершины 6-мерного гиперкуба. Проект исследует интеграцию 7 теоретических источников (Склярова, Фомюк, Андреев, Касаткин, Герман, Беляев, SOLAN) в единую архитектуру.

### Эволюция подходов к медиации между модулями

```
v53-v54:  Модули подключены напрямую к forward pass          → PPL 2.91 (vanilla-уровень)
v55-v56:  ConvergenceBridge + TernaryQuantizer                → Теоретический фундамент
v57:      AbrialeLayer (событийная активация)                 → Commit-gate механизм
v58:      BridgeOfModules (иерархическое дерево мостов)       → PPL 1.35 (прорыв!)
v59:      AbrialeBridgeMediator (Abriale + Bridge)            → PPL 1.24 (лучший результат)
v60:      ArchetypalInterlingua (hub-and-spoke, 64 архетипа)  → PPL 2.93 (≈ vanilla)
v61:      BridgedInterlingua (мосты + архетипы, двойная прослойка) → PPL 2.92
```

## 3. Что НЕ работает (текущие проблемы)

### 3.1. ПРОБЛЕМА: Интерлингва не помогает (PPL ≈ vanilla)

**Факт:** Все варианты интерлингвы (v60, v61) дают PPL 2.92-2.94, что статистически неотличимо от vanilla (2.94). При этом BridgeOfModules даёт 1.35, а AbrialeBridge — 1.24.

**Разрыв:**
```
Лучший bridge:        PPL 1.24  (AbrialeBridge)
Лучшая интерлингва:   PPL 2.92  (BridgedInterlingua lightweight)
vanilla:              PPL 2.94
                      ──────────────────────────────
Вклад интерлингвы:    -0.02 PPL  (= шум)
Вклад мостов:         -1.70 PPL  (= существенный)
```

### 3.2. ПРОБЛЕМА: Gate ≈ 0.5 — модель игнорирует интерлингву

Во всех конфигурациях global_gate стабилизируется на ~0.49, что означает:
- Модель смешивает основной путь и интерлингву примерно пополам
- Но это не "золотая середина" — это начальное значение, от которого gate не отклонился
- Если бы интерлингва была полезна, gate сдвинулся бы в её сторону (> 0.5)

```
Инициализация gate:   0.500
После 800 шагов:      0.489-0.494  (сдвиг ≈ -0.01, в сторону "не использовать")
```

### 3.3. ПРОБЛЕМА: Нет дифференциации тритов

| Конфиг | trit_pos | trit_zero | trit_neg | Интерпретация |
|--------|----------|-----------|----------|---------------|
| v60 interlingua | 0.0 | 1.0 | 0.0 | Все нули — полное воздержание |
| v61 bridged_lightweight | 1.0 | 0.0 | 0.0 | Все +1 — единогласное одобрение |
| v61 bridged_full | 0.0 | 1.0 | 0.0 | Все нули |
| interlingua_ternary (после фикса) | 0.0 | 0.0 | 1.0 | Все -1 — единогласное отрицание |
| interlingua_binary | 0.33 | 0.33 | 0.33 | Равномерное (без тернарной квантизации) |

**Диагноз:** Все модули «голосуют» одинаково за каждый архетип. Нет дифференциации = нет информации = readout attention видит одинаковые архетипы.

### 3.4. ПРОБЛЕМА: Предсказание P1 не подтверждено

**P1:** «Interlingua с N > 4 должна показать лучшую масштабируемость, чем Bridge.»

**Результат:** Interlingua (2.93) >> Bridge (1.35). Мосты в 2x лучше по PPL. P1 **не подтверждено** на 800 шагах обучения.

### 3.5. ПРОБЛЕМА: 800 шагов < 2000 warmup

Temperature annealing настроен на warmup_steps=2000 (по умолчанию). К шагу 800 температура снизилась лишь с 1.0 до 0.689 — модель ещё не дошла до «жёсткой» фазы (T → min_temp=0.1).

```
Фаза обучения:
  Шаг   1:    T = 1.000  (мягкая, все градиенты проходят, но триты бессмысленны)
  Шаг 200:    T = 0.978  (почти мягкая)
  Шаг 400:    T = 0.914  (ещё мягкая)
  Шаг 600:    T = 0.815  (средняя)
  Шаг 800:    T = 0.689  ← ЗДЕСЬ остановились
  Шаг 1200:   T = 0.397  (начинает затвердевать)
  Шаг 1600:   T = 0.191  (почти жёсткая)
  Шаг 2000:   T = 0.100  (жёсткая — триты {-1, 0, +1})
```

## 4. Корневой диагноз: информационный bottleneck

### Цепочка причин

```
1. Общий trit_proj для всех модулей
   ↓
2. Все модули проецируются через одну и ту же матрицу
   ↓
3. Триты получаются одинаковые для всех модулей
   ↓
4. Consensus = sign(N × one_trit) = тот же one_trit
   ↓
5. 64 архетипа все одинаковые
   ↓
6. Readout attention видит 64 одинаковых вектора
   ↓
7. Выход интерлингвы = avg(одинаковых) = бесполезный constant
   ↓
8. Gate учится к ≈0.5 (минимальный вред от бесполезного сигнала)
   ↓
9. PPL ≈ vanilla
```

### Ключевое отличие от Bridge

В BridgeOfModules (PPL 1.35):
- Каждая пара модулей обрабатывается **отдельным** мостом с **своими** параметрами
- Мосты видят **различия** между парами модулей
- Иерархия мостов постепенно объединяет информацию

В Interlingua (PPL 2.93):
- Все модули проходят через **один и тот же** trit_proj
- Архетипы — это shared anchors (Q6 вершины), но кодирование **не дифференцировано**
- Модель не может выразить «модуль A голосует за архетип #17, а модуль B — против»

## 5. Варианты решения (многие варианты)

### Вариант A: Longer Training (>2000 шагов)

**Идея:** Просто обучать дольше, дождаться жёсткой фазы temperature annealing.

| За | Против |
|----|--------|
| Самый простой вариант, нет изменений кода | 800 шагов уже показали тренд — gate идёт к 0.49, не к 0.5+ |
| Температура к шагу 2000 достигнет 0.1 | Даже с жёсткими тритами проблема одинаковости сохранится |
| Можно запустить параллельно с другими фиксами | Время: ~3x больше (2400 шагов = ~45 мин) |

**Оценка:** Необходимое, но недостаточное условие. Вероятность помочь: **20%**.

### Вариант B: Per-Source trit_proj (отдельный проектор для каждого модуля)

**Идея:** Заменить один общий `trit_proj` на N отдельных проекторов.

```python
# Текущая реализация (общий):
trit_proj = nn.Linear(d_model, n_archetypes)  # 1 штука для всех

# Предлагаемая (per-source):
trit_projs = nn.ModuleList([
    nn.Linear(d_model, n_archetypes) for _ in range(n_sources)
])
```

| За | Против |
|----|--------|
| Прямо решает проблему одинаковости тритов | +N параметров (но незначительно: N × d × 64) |
| Каждый модуль учится своему паттерну голосования | Может привести к хаотичному голосованию без регуляризации |
| Минимальные изменения кода (1 файл) | Нужен diversity loss для предотвращения коллапса |

**Оценка:** Высокая вероятность помочь. Вероятность: **60%**.

### Вариант C: Diversity Loss (штраф за одинаковое голосование)

**Идея:** Добавить регуляризацию, штрафующую ситуацию, когда все модули голосуют одинаково.

```python
# Diversity loss: минимизировать корреляцию между тритами разных модулей
trit_matrix = stack([trits_module_i for i in range(N)])  # shape: (N, 64)
cosine_sim = trit_matrix @ trit_matrix.T / (norms * norms.T)
diversity_loss = cosine_sim.mean() - cosine_sim.diag().mean()  # off-diagonal mean
```

| За | Против |
|----|--------|
| Явно поощряет дифференциацию | Ещё один гиперпараметр (вес diversity loss) |
| Работает с существующим общим trit_proj | Может конфликтовать с основной задачей (LM loss) |
| Теоретически обоснован (максимизация информации) | Не гарантирует полезную дифференциацию |

**Оценка:** Хорошее дополнение к варианту B. Вероятность: **40%** (solo), **70%** (с B).

### Вариант D: Learned Archetype Embeddings (вместо фиксированных Q6)

**Идея:** Вместо 64 фиксированных Q6-якорей использовать обучаемые эмбеддинги архетипов.

```python
# Текущий (фиксированный):
archetype_anchors = generate_q6_vertices()  # (64, 6), фиксированные

# Предлагаемый (обучаемый):
archetype_embeddings = nn.Parameter(torch.randn(64, d_model))  # обучаемые
```

| За | Против |
|----|--------|
| Архетипы адаптируются к задаче | Теряется связь с Q6 геометрией (одна из ключевых идей проекта) |
| Readout attention получает разнообразные якоря | Нужна инициализация из Q6 для сохранения bias |
| Увеличивает ёмкость модели | Может переобучиться на маленьких данных |

**Оценка:** Работает для PPL, но теоретически спорно. Вероятность: **50%**.

### Вариант E: Hybrid Bridge+Interlingua (сохранить мосты, добавить архетипы)

**Идея:** Не заменять мосты на интерлингву, а использовать параллельно.

```
Модуль₁ ─┬─ Bridge₁ ──────────┬── merge ──→ ядро
Модуль₂ ─┘                     │
Модуль₃ ─┬─ Bridge₂ ──────────┤
Модуль₄ ─┘                     │
...                             │
Все модули ─── Interlingua ────┘  (параллельный путь через 64 архетипа)
```

| За | Против |
|----|--------|
| Мосты уже работают (PPL 1.35) | Удвоение параметров медиации |
| Интерлингва добавляет интерпретируемость | Сложность архитектуры |
| Не теряем проверенный механизм | Интерлингва может быть проигнорирована (как сейчас) |

**Оценка:** Безопасный вариант. Вероятность улучшить PPL: **30%** (поверх Bridge).

### Вариант F: Contrastive Archetype Learning

**Идея:** Обучать архетипы контрастивно: разные модули должны давать разные паттерны активации архетипов для одного и того же токена.

```python
# Для каждого токена t:
#   - Модуль i активирует набор архетипов A_i(t)
#   - Модуль j активирует набор архетипов A_j(t)
#   - Loss: max(0, threshold - hamming(A_i(t), A_j(t)))
```

| За | Против |
|----|--------|
| Прямо принуждает к дифференциации | Сложная реализация |
| Использует свойство Хэмминг-расстояния Q6 | Может помешать согласованности голосования |
| Теоретически связан с contrastive_pairs() SOLAN | Нужен баланс с LM loss |

**Оценка:** Теоретически красиво, практически сложно. Вероятность: **35%**.

### Вариант G: Lean Model (удалить вредные модули)

**Идея:** Ablation v59 показала, что 5 из 7 модулей вредят. Оставить только Heisenberg + FlowerGAT.

```
Текущая:  7 модулей → мосты/интерлингва → ядро     (PPL 1.35, 1.4M params)
Lean:     2 модуля → простой gate → ядро             (ожидание: PPL < 1.07, ~1M params)
```

| За | Против |
|----|--------|
| Удаление шума от 5 вредных модулей | Отказ от мультимодульной архитектуры |
| Меньше параметров, быстрее обучение | Интерлингва/мосты теряют смысл при N=2 |
| v59 ablation предсказывает PPL < 1.07 | Не решает проблему интерлингвы |

**Оценка:** Лучший результат PPL, но не продвигает теорию. Вероятность PPL < 1.1: **85%**.

### Вариант H: Attention-Based Trit Voting (вместо линейной проекции)

**Идея:** Заменить `trit = linear(source_output)` на attention-based scoring.

```python
# Текущий (линейная проекция):
trits = sign(trit_proj(source_output))  # shape: (B, 64)

# Предлагаемый (attention-based):
archetype_queries = archetype_embeddings  # (64, d)
source_keys = source_proj(source_output)   # (B, T, d)
attention_scores = archetype_queries @ source_keys.T  # (B, 64, T)
trits = sign(attention_scores.mean(dim=-1))  # (B, 64)
```

| За | Против |
|----|--------|
| Архетипы "запрашивают" информацию у модулей | Больше параметров и вычислений |
| Контекстно-зависимое голосование | Усложняет отладку |
| Аналогично cross-attention в мостах (работает!) | Может деградировать к обычному attention |

**Оценка:** Умеренная сложность. Вероятность: **45%**.

## 6. Варианты развития (стратегические направления)

### Направление 1: «Починить интерлингву» (Варианты B+C+A)

**Цель:** Заставить Archetypal Interlingua работать на уровне Bridge (PPL < 1.4).

**План:**
1. Реализовать per-source trit_proj (Вариант B)
2. Добавить diversity loss (Вариант C)
3. Обучить 3000+ шагов (Вариант A)
4. Замерить дифференциацию тритов, PPL, gate

**Ресурсы:** ~2 дня кода, ~1 час обучения на конфигурацию.

**Риск:** Интерлингва может быть фундаментально ограничена bottleneck-архитектурой — 64 архетипа могут быть недостаточно выразительными.

### Направление 2: «Lean + Scale» (Вариант G + валидация)

**Цель:** Получить лучший PPL с минимумом архитектуры, затем масштабировать.

**План:**
1. Lean model: только Heisenberg + FlowerGAT (без мостов/интерлингвы)
2. Бенчмарк на полном WikiText-103
3. Масштабирование до d=256, d=512
4. Проверка на реальных задачах (не только PPL)

**Ресурсы:** ~1 день кода, но больше вычислений для масштабирования.

**Риск:** Отказываемся от мультимодульности, теряем уникальность проекта.

### Направление 3: «Гибрид» (Вариант E + B)

**Цель:** Сохранить мосты (работают) + добавить интерлингву как интерпретируемый слой.

**План:**
1. Основной путь: BridgeOfModules (PPL 1.35)
2. Параллельный путь: Interlingua с per-source trit_proj
3. Мягкий merge: bridge_output * α + interlingua_output * (1-α)
4. Интерлингва даёт интерпретируемость (64 архетипа × тритовые паттерны)

**Ресурсы:** ~2 дня.

**Риск:** Интерлингва остаётся декоративной (gate → 0).

### Направление 4: «SOLAN-first» (Вариант отдельный)

**Цель:** Сменить фокус с медиации модулей на интеграцию GlyphTokenizer.

**План:**
1. Подключить GlyphTokenizer вместо CharTokenizer
2. Бенчмарк: даёт ли визуальный Q6-изоморфизм преимущество
3. Если да — использовать глиф-энкодинг как вход для интерлингвы

**Ресурсы:** ~3 дня (интеграция нетривиальна).

**Риск:** SOLAN-76 = только 76 символов, не подходит для general-purpose LM.

### Направление 5: «Теория первая» (углубление формализации)

**Цель:** Понять, **почему** мосты работают, а интерлингва нет, через математический анализ.

**Ресурсы:** ~3-5 дней исследований.

## 10. Открытые вопросы

1. **Почему cross-attention в мостах работает, а в archetype readout — нет?**
   - Гипотеза: мосты получают **попарные** сигналы (2 входа), а readout — агрегированные (1 вход)
   - Проверка: добавить попарную информацию в архетипы

2. **Достаточно ли 800 шагов для справедливого сравнения?**
   - Мосты к шагу 200 уже начинают отрываться от vanilla
   - Интерлингва к шагу 800 ещё на уровне vanilla → вероятно, нет

3. **Является ли bottleneck в 64 архетипа фундаментальным ограничением?**
   - interlingua_128_archetypes (PPL 2.93) не лучше 64 → да, проблема не в числе архетипов
   - Проблема в кодировании, не в числе

4. **Нужна ли тернарная логика вообще?**
   - bridged_no_ternary (PPL 2.93) = bridged_lightweight (PPL 2.92) → разницы нет
   - Тернарность пока не проявила себя

5. **Совместимы ли мосты и интерлингва?**
   - v61 BridgedInterlingua = мосты + интерлингва, но PPL ≈ vanilla
   - Мосты в v61 тоже не работают → архетипальный слой «убивает» сигнал мостов
