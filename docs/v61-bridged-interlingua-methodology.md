# v61: BridgedInterlingua — Методология двойной прослойки

## 1. Проблема

Текущие архитектуры используют **одинарную прослойку** между модулями и ядром:

```
v58 (BridgeOfModules):     Модуль → Bridge → Bridge → ... → Ядро     O(N log N)
v60 (Interlingua):         Модуль → Encoder → 64 архетипа → Ядро     O(N)
```

**Недостатки v58**: дерево мостов сжимает N сигналов в 1 — информация теряется на каждом уровне.

**Недостатки v60**: каждый модуль кодируется **независимо** — нет попарной медиации.
Если два модуля дают конфликтующие сигналы, конфликт передаётся прямо в архетипы.

## 2. Решение: двойная прослойка

```
Модуль₁ ─┬─ Bridge₁ ─┐
Модуль₂ ─┘            │
                       ├── Encoder₁ ──┐
Модуль₃ ─┬─ Bridge₂ ─┘               │
Модуль₄ ─┘                           ├── 64 Архетипа → Readout → Ядро
                       ┌── Encoder₂ ──┘               │
Модуль₅ ─┬─ Bridge₃ ─┤                               │
Модуль₆ ─┘            └── Encoder₃ ──┘               │
                                                      │
Модуль₇ ──── OddProj ──── Encoder₄ ─────────────────┘
```

**Порядок подключения:**
1. Сначала **мосты подключаются к модулям** — снимают интерференцию
2. Затем bridge-выходы **подключаются к 64 архетипам** — единая семантика
3. Архетипы **подключаются к ядру** через readout cross-attention

## 3. Методология сочетания мостов и архетипов

### 3.1. Правила спаривания модулей (какие мосты к каким модулям)

Текущее спаривание — по порядку регистрации:
- Bridge₁: Heisenberg + Palace (attention-based пара)
- Bridge₂: PrivilegedAxis + FlowerGAT (geometry-based пара)
- Bridge₃: CubeDiagonal + D4Equivariant (symmetry-based пара)
- Odd: DualEmbedding (проходит через отдельную проекцию)

**Почему это работает**: мосты соединяют модули из смежных доменов,
где вероятность конструктивной интерференции выше.

### 3.2. Один уровень мостов, не дерево

В отличие от v58 (полное дерево), v61 использует **один уровень** мостов:
- Пары модулей → K = ceil(N/2) bridge-выходов
- Без рекурсивного сжатия → меньше потерь информации
- K bridge-выходов все идут в архетипальный слой параллельно

### 3.3. Архетипальный слой получает "очищенные" сигналы

После мостовой медиации:
- Деструктивная интерференция уже снята мостами
- Каждый bridge-выход — согласованное представление пары
- Архетипальный слой видит 4 сигнала вместо 7 → проще агрегировать
- Тернарное голосование точнее (меньше шума)

### 3.4. Тернарная агрегация по bridge-выходам

Каждый bridge-выход голосует за каждый из 64 архетипов:
- **+1** (ян): сильная поддержка архетипа
- **0** (变爻): неопределённость / воздержание
- **-1** (инь): отрицание архетипа

Консенсус: `sign(sum(trits_all_bridges))` → итоговый голос

## 4. Теоретические предсказания

| # | Гипотеза | Механизм |
|---|----------|----------|
| H1 | BridgedInterlingua < v60 по PPL | Мосты фильтруют конфликты ДО архетипов |
| H2 | BridgedInterlingua < v58 по PPL | Архетипы дают единый язык вместо сжатия дерева |
| H3 | Ternary помогает в двойной прослойке | После мостов сигналы чище → триты точнее |
| H4 | Overhead < 30% по времени vs v60 | 1 уровень lightweight мостов = мало параметров |

## 5. Архитектурные параметры

| Параметр | Значение | Обоснование |
|----------|----------|-------------|
| `bridge_mode` | 'lightweight' | Bilinear быстрее cross-attention, достаточен для 1 уровня |
| `n_archetypes` | 64 | Гексаграммы И-Цзин, Q6 якоря |
| `use_ternary` | True | Тернарная логика Аймара: {-1, 0, +1} |
| `uncertainty_budget` | 0.3 | 30% голосов — "воздержание" |
| `n_heads` | 4 | Для encode_attn и readout_attn |

## 6. Сложность

| Компонент | Сложность | Параметры |
|-----------|-----------|-----------|
| Мосты (1 уровень) | O(N/2) | N/2 × LightweightBridge |
| Encoders | O(K), K=ceil(N/2) | K bottleneck encoders |
| Encode cross-attn | O(K × A × T) | A=64 архетипа, T=длина |
| Ternary aggregation | O(K × A) | Линейная |
| Readout cross-attn | O(T × A) | Одна операция |
| **Итого** | **O(N + A × T)** | ~O(N) для фикс. T |

## 7. API совместимость

`BridgedInterlingua` реализует тот же интерфейс, что `ArchetypalInterlingua`:
- `forward(x, source_outputs)` → enriched
- `get_interlingua_loss()` → aux loss
- `get_interlingua_stats()` → diagnostics dict
- `archetype_q6_correlation()` → Q6 correlation

Активируется флагом `use_bridged_interlingua=True` в конфиге.
При этом `use_archetypal_interlingua` автоматически устанавливается в True
для переиспользования forward path в model.py.
