## СТУПЕНЬ 1: Склярова — «Компьютер Древнего Китая»

### Что берём

**Идея:** И-Цзин — это не гадательная система, а **вычислительный механизм** с конкретной бинарной архитектурой.

**Конкретные элементы для интеграции:**

| Элемент | Объём | Где применяется | Статус |
|---------|-------|-----------------|--------|
| Триграммы как 3-битные регистры (вход/выход) | Уже реализовано | `geometry.py` | ✅ Есть |
| Гексаграмма = пара регистров (верх × низ) | Уже реализовано | `geometry.py` | ✅ Есть |
| Порядок Вэнь-вана vs Фуси как два кодирования | Новый код: ~50 строк | `geometry.py` | ❌ Нет |
| Восемь дворцов (宮) как 8 кластеров по 8 | Новый код: ~80 строк | `geometry.py` | ❌ Нет |

**Что нового даёт:**

Склярова обосновывает, что порядок Фуси (= binary counting) и порядок Вэнь-вана (= empirical ordering) — это **два разных кодирования одного пространства**. В терминах ML:
- Порядок Фуси → **natural binary code** (Gray code) — оптимален для квантизации
- Порядок Вэнь-вана → **learned permutation** — оптимален для семантики

**Действия:**

1. Добавить в `geometry.py` функцию `fuxi_order()` и `wenwang_order()` — два канонических порядка 64 гексаграмм
2. Добавить `palace_clusters()` — 8 дворцов по 8 гексаграмм (группировка по Вэнь-вану)
3. Добавить опцию `codebook_order` в конфиг: `fuxi` | `wenwang` | `learned`

**Обоснование новизны:** В текущем v50 используется только один порядок ({-1,+1}⁶ в лексикографическом). Два канонических порядка + обучаемая перестановка — это **три режима работы кодбука**, чего нет ни в одном существующем VQ-подходе.

## СТУПЕНЬ 2: Фомюк — Антиподальность и Ло-шу

### Что берём

**Идея:** Каждая гексаграмма имеет **антипод** (инверсию всех линий). Пары антиподов — фундаментальная структура, задающая «баланс» кодбука.

**Конкретные элементы:**

| Элемент | Объём | Где применяется | Статус |
|---------|-------|-----------------|--------|
| Антиподальное спаривание: hex(i) + hex(63-i) = const | Новый код: ~30 строк | `geometry.py` | ❌ Нет |
| Октетная структура (8 групп по 8) | Пересекается со Ступенью 1 | `geometry.py` | ❌ Нет |
| Ло-шу 3×3 как attention-паттерн | Новый код: ~40 строк | `geometry.py` | ❌ Нет |
| Антиподальная регуляризация в loss | Новый код: ~20 строк | `training/regularization.py` | ❌ Нет |

**Что нового даёт:**

Фомюк доказывает, что антиподальные пары обладают свойством **постоянной суммы**: для любой метрики d(hex_i) + d(hex_{63-i}) = const. Это означает:

```
Если кодбук квантизатора "знает" значение hex_i,
он автоматически "знает" значение hex_{63-i} = NOT(hex_i).
```

В терминах ML это **удвоение эффективной ёмкости**: 64 кодовых слова работают как 128, потому что каждое слово несёт информацию и о своём антиподе.

**Действия:**

1. Добавить `AntipodalPairIndex` в `geometry.py`:
   ```python
   def antipodal_pairs():
       """32 пары (i, 63-i) для гексаграмм."""
       return [(i, 63 - i) for i in range(32)]
   ```

2. Добавить `AntipodalRegularization` в `training/regularization.py`:
   ```python
   # Штраф: ||Q(x) + Q(NOT(x)) - target_sum|| → 0
   # Заставляет квантизатор "знать" об антиподальности
   ```

3. Добавить `LoShuAttentionBias` — матрица 3×3 Ло-шу как микро-attention-паттерн:
   ```
   2 7 6     Ло-шу: сумма по строкам,
   9 5 1  →  столбцам и диагоналям = 15
   4 3 8     = магический квадрат 3×3
   ```
   Используется как инициализация весов для 3×3 свёрточного attention-bias.

**Обоснование новизны:** Ни VQ-VAE, ни VQGAN, ни Lila-E8 не используют антиподальную структуру кодбука. Это **новый вид структурной регуляризации**, эксплуатирующий симметрию Z₂⁶.

## СТУПЕНЬ 3: Андреев — Треугольная матрица

### Что берём

**Идея:** 64 гексаграммы можно расположить в **треугольной матрице**, где позиция определяется треугольным числом T(n) = n(n-1)/2 mod 64. Это задаёт нетривиальный порядок обхода.

**Конкретные элементы:**

| Элемент | Объём | Где применяется | Статус |
|---------|-------|-----------------|--------|
| Треугольная матрица 8×8 (нижнетреугольная) | Новый код: ~40 строк | `geometry.py` | ❌ Нет |
| Triangular attention mask | Новый код: ~60 строк | `models/model.py` | ❌ Нет |
| Curriculum ordering по треугольным числам | Новый код: ~30 строк | `training/` | ❌ Нет |
| Андреевский порядок обхода 64 гексаграмм | Новый код: ~20 строк | `geometry.py` | ❌ Нет |

**Что нового даёт:**

Андреев показывает, что треугольная матрица задаёт **неравномерную связность**: гексаграммы, расположенные ближе к диагонали, «сильнее связаны» друг с другом. Это в точности **attention bias**:

```
attention_bias[i][j] = f(треугольное_расстояние(hex_i, hex_j))
```

В отличие от стандартного positional encoding (линейное расстояние), треугольное расстояние учитывает **комбинаторную структуру** пространства гексаграмм.

**Действия:**

1. Реализовать `triangular_distance_matrix(n=64)`:
   ```python
   def triangular_distance(i, j, P=64):
       """Расстояние по Андрееву: через треугольные числа."""
       pos_i = (i * (i - 1) // 2) % P
       pos_j = (j * (j - 1) // 2) % P
       return min(abs(pos_i - pos_j), P - abs(pos_i - pos_j))
   ```

2. Добавить `TriangularAttentionBias` — новый attention bias на основе расстояний Андреева. Это альтернатива ALiBi и RoPE, использующая структуру гиперкуба.

3. Добавить `curriculum_order_andreev()` — порядок подачи данных при обучении, основанный на треугольных числах (сначала «простые» гексаграммы у диагонали, потом «сложные»).

**Обоснование новизны:** Существующие positional encoding (sinusoidal, RoPE, ALiBi) используют **линейное** расстояние между позициями. Треугольный bias — это **нелинейное** позиционное кодирование с математическим обоснованием из теории комбинаторики. Аналогов в литературе нет.

## СТУПЕНЬ 4: Касаткин — Диофантов куб

### Что берём

**Идея:** 64 гексаграммы можно координатизовать не как плоские 6-битные вектора, а как точки **трёхмерной целочисленной решётки** (диофантовы координаты). Каждая гексаграмма получает координату (x, y, z) ∈ Z³.

**Конкретные элементы:**

| Элемент | Объём | Где применяется | Статус |
|---------|-------|-----------------|--------|
| Диофантово отображение: hex → (x,y,z) ∈ Z³ | Новый код: ~60 строк | `geometry.py` | ❌ Нет |
| Кубическое 3D embedding вместо 6D | Новый код: ~80 строк | `models/model.py` | ❌ Нет |
| Ось Z как «ось времени» (линейная координата) | Новый код: ~30 строк | `models/model.py` | ❌ Нет |
| Диофантовы расстояния для attention | Новый код: ~40 строк | `geometry.py` | ❌ Нет |

**Что нового даёт:**

Текущая архитектура v50 использует 6-мерное представление: hex = (b₁, b₂, b₃, b₄, b₅, b₆) ∈ {-1,+1}⁶. Касаткин предлагает **альтернативное 3-мерное** представление:

```
6D (текущее):    hex → {-1,+1}⁶     (равноправные биты)
3D (Касаткин):   hex → (x, y, z)     (иерархические координаты)
```

Преимущество 3D: **визуализируемость** и **геометрическая интуиция**. В 3D можно:
- Визуализировать attention как потоки в кубе
- Определить «ось времени» (диагональ куба) = привилегированное направление
- Использовать евклидово расстояние в Z³ вместо расстояния Хэмминга в Z₂⁶

**Действия:**

1. Реализовать `kasatkin_embedding(hex_index) → (x, y, z)`:
   Отображение 64 гексаграмм в Z³ по правилам Касаткина.

2. Добавить `CubicPositionalEncoding` — позиционное кодирование, где каждый токен получает 3D-координату в кубе (вместо линейной позиции).

3. Добавить `CubicAttentionBias` — attention bias на основе евклидова расстояния в Z³:
   ```python
   bias[i][j] = -alpha * ||kasatkin(i) - kasatkin(j)||²
   ```

4. Добавить визуализацию: 3D-рендер куба с attention-потоками (для интерпретируемости).

**Обоснование новизны:** Все существующие трансформеры работают с **линейными** позициями (1D sequence). Касаткинское 3D-embedding помещает токены в **кубическую решётку**, что позволяет attention работать в трёх измерениях одновременно. Это уникальный подход, не имеющий аналогов в архитектурах трансформеров.

## СТУПЕНЬ 5: Герман — Теория упаковок P=2^k

### Что берём

**Идея:** Из всех замкнутых полей **только поля с периодом P = 2^k** допускают бесколлизионную упаковку по алгоритму треугольных чисел. Это **единственное математическое доказательство** того, почему именно степени двойки являются оптимальными размерами кодбука.

**Конкретные элементы:**

| Элемент | Объём | Где применяется | Статус |
|---------|-------|-----------------|--------|
| Доказательство: только P=2^k → без коллизий | Документация: ~2 страницы | `theoretical-foundations.md` | ❌ Нет |
| Упаковочный алгоритм (АУ) | Новый код: ~40 строк | `geometry.py` | ❌ Нет |
| Антиподальная теорема: n + n̄ = P+1 | Новый код: ~20 строк | `geometry.py` | ❌ Нет |
| Формула коллизий: (n₁+n₂)(n₂-n₁+1)/2P = q | Новый код: ~30 строк | `geometry.py` | ❌ Нет |

**Что нового даёт:**

В текущем v50 размер кодбука = 64 = 2⁶ выбран **эмпирически** (потому что так в И-Цзин). Герман даёт **строгое доказательство** оптимальности:

> **Теорема Германа.** Замкнутое клеточное поле периода P допускает полную бесколлизионную упаковку по алгоритму АУ тогда и только тогда, когда P = 2^k для некоторого k ∈ N.

Это означает:
- P = 64 = 2⁶ → **оптимально** (доказано)
- P = 128 = 2⁷ → **оптимально** (для расширенного кодбука)
- P = 60, 81, 100 → **не работают** (доказано, что возникнут коллизии)

**Следствие для архитектуры:** При масштабировании кодбука допустимы ТОЛЬКО размеры 2^k:
```
8 → 16 → 32 → 64 → 128 → 256 → 512 → 1024 → 4096
```

Любые другие размеры (например, 240 у E8) **гарантированно** имеют коллизии при циклической упаковке.

**Действия:**

1. Добавить **Теорему 8** в `theoretical-foundations.md`:
   - Формулировка теоремы Германа
   - Ссылка на оригинальную работу
   - Следствия для выбора размера кодбука

2. Реализовать `hermann_packing(P)` — алгоритм упаковки:
   ```python
   def hermann_packing(k):
       """Бесколлизионная упаковка P=2^k по алгоритму Германа."""
       P = 2 ** k
       field = [0] * P
       pos = 0
       for n in range(1, P + 1):
           field[pos % P] = n
           pos += n  # треугольное смещение
       return field
   ```

3. Реализовать `collision_test(P)` — верификация: для P ≠ 2^k найти коллизию, для P = 2^k доказать отсутствие.

4. Добавить `ValidCodebookSizes` — список допустимых размеров кодбука с математическим обоснованием.

**Обоснование новизны:** Ни один существующий VQ-подход (VQ-VAE, VQGAN, RQ-VAE, Lila-E8) **не обосновывает выбор размера кодбука теоретически**. Обычно берут 256, 512, 1024, 4096, 8192 по эмпирическим соображениям. Теорема Германа — **первое строгое обоснование** того, что именно степени двойки оптимальны для циклических кодбуков. Это уникальный теоретический результат.

## СТУПЕНЬ 6: Беляев — Куб Закона (Рычажная архитектура)

### Что берём

**Идея:** Принцип дополнительности (Бор, Гейзенберг) реализуется через **рычажные весы** — структуру, где прямая и обратная функции уравновешивают друг друга. Куб = объединение двух ортогональных крестов. Гиперкуб = иерархия кубов.

**Конкретные элементы:**

| Элемент | Объём | Где применяется | Статус |
|---------|-------|-----------------|--------|
| Рычажное уравнение: (a/b)^{+1} = -(b/a)^{-1} | Новый код: ~50 строк | `models/model.py` | ❌ Нет |
| Дуадная система координат (антагонисты) | Новый код: ~40 строк | `geometry.py` | ❌ Нет |
| Триадная система координат (сопряжённые) | Новый код: ~40 строк | `geometry.py` | ❌ Нет |
| Dual-cross attention (два креста) | Новый код: ~100 строк | `models/model.py` | ❌ Нет |
| Цветок Жизни — топология связей гиперкуба | Новый код: ~80 строк | `geometry.py` | ❌ Нет |

**Что нового даёт:**

Беляев даёт **операционную** интерпретацию кубической геометрии. Не просто «64 точки в пространстве», а **динамическая система** с рычажным механизмом:

```
Стандартный attention:
  score = Q · K^T / √d        (одностороннее)

Рычажный attention Беляева:
  score_direct  = (Q/K)^{+1}   (прямой рычаг)
  score_inverse = -(K/Q)^{-1}   (обратный рычаг)
  score = balance(score_direct, score_inverse)
```

Два креста (дуадный и триадный) дают **два типа связей** между вершинами куба:
- Дуадный: ⟨X|Y|Z⟩ ↔ ⟨-X|-Y|-Z⟩ (антиподы = антагонисты)
- Триадный: ⟨X|X|Y⟩ ↔ ⟨Y|Z|Z⟩ (соседи = сопряжённые)

**Действия:**

1. Реализовать `LeverageAttention` — рычажный attention:
   ```python
   class LeverageAttention(nn.Module):
       """
       Attention по принципу рычажных весов Беляева.
       Прямая + обратная компоненты уравновешивают друг друга.
       """
       def forward(self, q, k, v):
           # Прямой рычаг: Q доминирует
           direct = (q @ k.T) * self.scale
           # Обратный рычаг: K доминирует
           inverse = -(k @ q.T).T * self.scale
           # Баланс (обучаемый)
           alpha = torch.sigmoid(self.balance_param)
           scores = alpha * direct + (1 - alpha) * inverse
           return softmax(scores) @ v
   ```

2. Реализовать `DualCrossArchitecture` — два ортогональных «креста» в каждом слое:
   - Крест 1 (дуадный): attention между антиподами
   - Крест 2 (триадный): attention между соседями
   - Результат: weighted sum, формирующий «куб» на каждом слое

3. Реализовать `FlowerOfLifeTopology` — граф связей гиперкуба по Цветку Жизни:
   - Центральные кубики (2 шт.) = внутренний контур
   - Периферийные кубики (6 шт.) = внешний контур
   - Лепестки Мёбиуса = порядок обхода

4. Добавить `HypercubeLevelHierarchy` — 4 уровня иерархии:
   ```
   Уровень I:   2 элемента (инь/ян)      — binary gate
   Уровень II:  8 триграмм              — 3-bit quantizer
   Уровень III: 64 гексаграммы           — 6-bit quantizer (текущий)
   Уровень IV:  4096 двойных гексаграмм  — 12-bit quantizer (новый)
   ```

**Обоснование новизны:** Рычажный attention — **принципиально новый механизм**, не имеющий аналогов в литературе. Стандартный attention вычисляет Q·K^T (= «вопрос ищет ответ»). Рычажный добавляет обратную компоненту K·Q^T (= «ответ ищет вопрос»), уравновешивая их. Это реализует принцип дополнительности Бора на уровне архитектуры нейросети.

## СТУПЕНЬ 1 (расширение): Склярова

### 1.1. Лейбниц и двоичное представление (историческое обоснование)

**Что:** Склярова показывает, что Лейбниц (1703) увидел в гексаграммах двоичную систему счисления. Это не просто исторический факт — это **первый документированный случай** распознавания структуры {0,1}^n в культурном артефакте.

**Для чего в коде:** Добавить в документацию `theoretical-foundations.md` как Теорему 0 (Лейбниц-И-Цзин изоморфизм). Это укрепляет историческую легитимность проекта и показывает, что идея {-1,+1}^n не «натянутая», а имеет 320-летнюю историю.

**Объём:** ~30 строк документации. Чистая теория, кода нет.

### 1.2. Четыре состояния линий (老阳, 少阳, 少阴, 老阴)

**Что:** В традиции каждая линия имеет не 2, а **4 состояния**: старый ян (+1, меняющийся), молодой ян (+0.5, стабильный), молодая инь (-0.5, стабильная), старая инь (-1, меняющаяся). Это даёт 4⁶ = 4096 состояний вместо 2⁶ = 64.

**Для чего в коде:** Это расширение кодбука с 64 до 4096 БЕЗ увеличения размерности (всё ещё 6D, но в {-1, -0.5, +0.5, +1}⁶). Уже упомянуто в `yijing-transformer-concept.md` (раздел 6.2), но НЕ реализовано.

**Объём:** ~80 строк кода. Новый квантизатор `FourStateQuantizer` с 4 уровнями вместо 2.

**Техническая значимость:** Это **Product Quantization** с M=6 подпространствами по K=4 центроида = стандартная техника VQ, но с фиксированными центроидами из И-Цзин. Аналог есть в VQ-литературе (PQ), но конкретная геометрия {-1, -0.5, +0.5, +1} — нова.

### 1.3. Операция 變 (Бянь) — мутация линий как оператор

**Что:** В И-Цзин «старые» линии мутируют (ян → инь, инь → ян). Это **однозначно определённый оператор** на пространстве гексаграмм: каждая «старая» линия переключается, «молодые» остаются.

**Для чего в коде:** Текущий `BianGuaTransform` (v50) делает мягкую инверсию всех координат с обучаемыми вероятностями. Но в оригинале мутируют ТОЛЬКО «старые» линии — те, что в экстремальных позициях ±1. Линии в ±0.5 стабильны. Это даёт **градуированный** BianGua.

**Объём:** ~50 строк — модификация существующего `BianGuaTransform`.

### 1.4. Восемь дворцов (八宮) как attention-кластеры

**Что:** 64 гексаграммы делятся на 8 «дворцов» по 8 штук. Каждый дворец объединён общей нижней триграммой (или «корневой» гексаграммой по Вэнь-вану). Это **иерархическая кластеризация** гиперкуба.

**Для чего в коде:** Новый `PalaceClusterAttention` — attention внутри дворцов (intra-palace) сильнее, чем между дворцами (inter-palace). Это **structured sparsity** для attention-матрицы:

```
Полный attention 64×64 → блочный 8×(8×8)
Экономия: O(64²) → O(8 × 8²) = O(512) vs O(4096) = 8× ускорение
```

**Объём:** ~100 строк. Новый модуль `PalaceAttention`.

**Техническая значимость:** Это аналог **block-sparse attention** (BigBird, Longformer), но блоки определены не произвольно, а по математической структуре гексаграмм.

## СТУПЕНЬ 2 (расширение): Фомюк

### 2.1. Магический квадрат 3×3 как свёрточное ядро

**Что:** Ло-шу — это не просто «магический квадрат 3×3». Он обладает свойством: все строки, столбцы и диагонали дают сумму 15. Более того, разность между противоположными элементами относительно центра (5) постоянна.

**Для чего в коде:** Матрица Ло-шу как **фиксированное свёрточное ядро 3×3** для обработки триграмм. В каждом триграммном представлении 3 бита → развёрнутые в квадрат 3×3, свёрнуты с ядром Ло-шу:

```python
LOSHU_KERNEL = torch.tensor([
    [2, 7, 6],
    [9, 5, 1],
    [4, 3, 8]
], dtype=torch.float32) / 15.0  # нормализация
```

**Объём:** ~30 строк. Добавка в `geometry.py` + использование в `model.py`.

### 2.2. Октетная симметрия и группа D₄

**Что:** 8 триграмм образуют группу диэдра D₄ (симметрии квадрата): 4 поворота + 4 отражения. Это означает, что любое преобразование триграммы сводится к одной из 8 операций D₄.

**Для чего в коде:** `D4EquivariantLayer` — слой, эквивариантный относительно группы D₄. Аналог group-equivariant CNN (Cohen & Welling, 2016), но для дискретной группы D₄ на триграммах.

**Объём:** ~120 строк. Новый модуль.

**Техническая значимость:** Group-equivariant сети — передний край ML (e-CGNN, SE(3)-Transformers). D₄-эквивариантность для И-Цзин — **конкретный случай** общей теории, с прямой связью к октетной модели Фомюка.

### 2.3. Свойство постоянной суммы как constraint

**Что:** Фомюк показывает: для ЛЮБОЙ пары антиподов сумма их «весов» (по любой линейной метрике) постоянна. Формально: ∀f линейная, f(hex_i) + f(hex_{63-i}) = const.

**Для чего в коде:** Это **жёсткий constraint** на обучаемые эмбеддинги кодбука. Если мы выучили embedding для hex_i, embedding для hex_{63-i} ОПРЕДЕЛЁН автоматически. Это:
- Сокращает число параметров вдвое (64 → 32 свободных эмбеддинга)
- Гарантирует антиподальную когерентность
- Является формой **weight tying** (аналогия: shared embeddings input/output)

**Объём:** ~40 строк. Модификация `YiJingQuantizer`.

### 2.4. Магический квадрат 4×4 из упаковки Германа (перекрёстная связь)

**Что:** Герман (Ступень 5) показал, что при P=16=4² его упаковка порождает массив, из которого **перестановками внутри столбцов** получается магический квадрат 4×4. Фомюк показывает аналогичное для 3×3. Вместе: **магические квадраты любого порядка 2^k × 2^k** конструируются из упаковки Германа.

**Для чего в коде:** `MagicSquareInitializer` — инициализация весов attention-матрицы магическим квадратом соответствующего порядка. Для 8 голов attention: 8 магических квадратов 8×8.

**Объём:** ~60 строк. Функция в `geometry.py` + использование как инициализация.

## СТУПЕНЬ 3 (расширение): Андреев

### 3.1. Четыре уровня треугольной матрицы

**Что:** Андреев организует 64 гексаграммы в 4-уровневую иерархию по треугольным числам:
- Уровень 1: T(1)=0, T(2)=1 → позиции 0, 1
- Уровень 2: T(3)=3, T(4)=6 → позиции 3, 6
- Уровень 3: T(5)=10, ..., T(8)=28 → позиции 10, 15, 21, 28
- Уровень 4: T(9)=36, ..., T(64)=... → остальные

Каждый уровень — «слой» треугольной пирамиды.

**Для чего в коде:** `HierarchicalTriangularEncoding` — 4-уровневое позиционное кодирование, где каждый токен получает свой уровень в треугольной иерархии. Attention между уровнями ослабляется (или усиливается) — это **иерархический attention bias**.

**Объём:** ~70 строк.

### 3.2. Связь треугольных чисел с алгоритмом Германа

**Что:** Упаковочный алгоритм (АУ) Германа размещает число n в позицию T(n) = n(n-1)/2 mod P. Это **в точности** треугольная матрица Андреева. Два автора пришли к одной формуле независимо.

**Для чего в коде:** Единая функция `triangular_position(n, P)`, используемая и в `hermann_packing()`, и в `andreev_matrix()`. Общая математическая база — одна реализация.

**Объём:** ~20 строк (рефакторинг).

**Техническая значимость:** Совпадение формул из двух независимых работ — сильное подтверждение фундаментальности треугольных чисел для циклических полей P=2^k. Это не артефакт одного автора.

### 3.3. Инверсный обход (reverse triangular order)

**Что:** Если прямой обход по треугольным числам T(n) = n(n-1)/2 даёт порядок 0, 1, 3, 6, 10, 15, 21, 28, ..., то обратный обход T'(n) = P - T(P+1-n) даёт **антиподальный порядок**. Прямой и обратный порядки — это две «стороны» ленты Мёбиуса.

**Для чего в коде:** `bidirectional_triangular_walk()` — два attention-паттерна (прямой и обратный), подобно bidirectional attention в BERT, но с треугольной геометрией.

**Объём:** ~40 строк.

### 3.4. Curriculum learning по треугольным уровням

**Что:** Андреевская иерархия задаёт **естественный порядок обучения**: сначала обучаем на гексаграммах уровня 1 (самые связанные), потом постепенно добавляем уровни 2, 3, 4.

**Для чего в коде:** `TriangularCurriculum` — расписание обучения:
- Эпохи 1–10: только уровень 1 (2 гексаграммы)
- Эпохи 11–20: уровни 1+2 (4 гексаграммы)
- Эпохи 21–40: уровни 1+2+3 (8 гексаграмм)
- Эпохи 41+: все 64

Это аналог **curriculum learning** (Bengio et al., 2009), но с математически обоснованным порядком сложности.

**Объём:** ~50 строк в `training/`.

## СТУПЕНЬ 4 (расширение): Касаткин

### 4.1. Привилегированная ось (диагональ куба)

**Что:** Касаткин выделяет ось (0,0,0)→(1,1,1) как «ось Z» — **привилегированное направление** в кубе. В терминах И-Цзин это ось ☷(Земля)→☰(Небо). Все остальные 6 вершин вращаются вокруг этой оси.

**Для чего в коде:** `PrivilegedAxisAttention` — attention вдоль привилегированной оси сильнее, чем перпендикулярно. Это реализует идею «главной диагонали» как оси времени:

```python
# Проекция на ось (1,1,1)/√3
axis_proj = (q[..., 0] + q[..., 1] + q[..., 2]) / sqrt(3)
# Усиление attention вдоль оси
axis_bias = axis_proj_q * axis_proj_k * learnable_scale
```

**Объём:** ~40 строк.

### 4.2. Два типа диагоналей: главная и побочные

**Что:** В кубе 4 диагонали: одна главная (0,0,0)↔(1,1,1) и три побочные. Касаткин показывает, что главная диагональ соединяет **комплементарные** вершины (☷↔☰), а побочные — **частично комплементарные** (☳↔☴, ☵↔☲, ☱↔☶).

**Для чего в коде:** 4 типа attention-связей с разными весами:
- Главная диагональ: полная инверсия (все 3 бита)
- Побочные диагонали: частичная инверсия (1 или 2 бита)
- Рёбра: соседи (отличие в 1 бит)
- Грани: планарные соседи (отличие в 2 бита)

```python
DIAGONAL_TYPES = {
    'main': [(0, 7)],              # 3 бита различия
    'secondary': [(1, 6), (2, 5), (3, 4)],  # 3 бита
    'edge': 12 пар,                 # 1 бит различия
    'face': 12 пар,                 # 2 бита различия
```

**Объём:** ~60 строк.

### 4.3. Диофантово уравнение n² + 3n - 2X - 2Pq = 0

**Что:** Герман (Ступень 5) выводит уравнение (4): n² + 3n - 2X - 2Pq = 0, где n — число в клетке, X — номер клетки, q — число полных оборотов. Касаткин решает АНАЛОГИЧНЫЕ диофантовы уравнения для координат куба.

**Для чего в коде:** `DiophantineSolver` — решатель диофантовых уравнений для нахождения координат гексаграмм. Общий инструмент для обоих подходов (Герман + Касаткин).

**Объём:** ~80 строк.

**Техническая значимость:** Диофантовы уравнения — это **целочисленное программирование**. В ML это связано с quantization-aware training: нахождение ближайшей целочисленной точки на поверхности гиперкуба.

### 4.4. Dual embedding: 6D || 3D

**Что:** Вместо выбора «или 6D, или 3D» можно использовать **оба представления одновременно**:

**Для чего в коде:** `DualEmbedding` — каждый токен имеет два представления, связанные обучаемым проектором:

### 5.2. Формула коллизий и «почему не E8»

**Что:** Формула (2) Германа: (n₁ + n₂)(n₂ - n₁ + 1) / 2P = q (целое) — условие коллизии. Для P = 240 (E8) это выражение МОЖЕТ быть целым при конкретных n₁, n₂. Значит, E8-кодбук из 240 точек **НЕ свободен от коллизий** при циклической упаковке.

**Для чего в коде:** `e8_collision_finder(P=240)` — скрипт, находящий конкретные пары (n₁, n₂), создающие коллизию в E8. Это **сильный аргумент** против E8 и в пользу {-1,+1}^n.

**Объём:** ~40 строк.

**Техническая значимость:** Это **первое формальное доказательство** дефекта E8-кодбука при циклическом использовании. В статье Lila-E8 этот вопрос не рассматривался. Если коллизии подтвердятся — это публикуемый результат.

### 5.3. Следствие 1: Самосовпадение (fixed-point property)

**Что:** Герман доказывает: при ЛЮБОМ выборе начальной клетки, найдётся клетка, содержащая число, равное своему порядковому номеру. То есть ∃ n: position(n) = n.

**Для чего в коде:** `find_fixed_points(P)` — нахождение неподвижных точек для кодбука размера P. Неподвижные точки — это **якорные элементы** кодбука, не требующие переиндексации. Потенциально полезно для:
- Инициализации эмбеддингов (якорные точки фиксированы)
- Curriculum learning (начинать с неподвижных точек)

**Объём:** ~30 строк.

### 5.4. Следствие 2: Антиподальная теорема (n + n̄ = P+1)

**Что:** Герман ДОКАЗЫВАЕТ (не просто наблюдает, как Фомюк): сумма чисел в клетках, отстоящих на P/2, постоянна и равна P+1.

**Что:** При P = 2^(2k) (например, P=16=4²) упаковка Германа записывается в квадрат (2^k)×(2^k). Сумма по столбцам постоянна (следствие 2). Перестановками внутри столбцов получается магический квадрат.

## СТУПЕНЬ 6 (расширение): Беляев

### 6.1. Принцип неопределённости Гейзенберга → Attention

**Что:** Беляев переформулирует ΔxΔp ≥ ℏ/2 как рычажное уравнение:
(Δx→max / (ℏ/2)Δx→min)^{+1} = -((ℏ/2)Δp→max / Δp→min)^{-1}

Это означает: **точность «вопроса» и точность «ответа» обратно пропорциональны**.

**Для чего в коде:** В attention Q·K^T/√d_k, параметр d_k играет роль ℏ/2 — он задаёт «гранулярность» скалярного произведения. Беляев даёт ФИЗИЧЕСКОЕ обоснование формулы масштабирования attention. Новая формула:

```python
# Стандартный: score = Q·K^T / sqrt(d_k)
# Беляевский: score = (Q·K^T)^{+1/T} / (K·Q^T)^{-1/T}
# где T — "температура" = аналог ℏ
```

**Объём:** ~50 строк.

### 6.2. «Песочный крест» — 4 квадранта attention

**Что:** Два рычага (прямой и обратный), каждый с двумя компонентами (числитель и знаменатель) = 4 квадранта. Беляев визуализирует это как «песочный крест» из 4 песочных часов.

**Для чего в коде:** `QuadrantAttention` — attention с 4 квадрантами:

```
         max(Q)
           |
  min(K) ──●── max(K)
           |
         min(Q)

Квадрант I:   max(Q), max(K) → сильный positive attention
Квадрант II:  max(Q), min(K) → сильный negative (контрастный)
Квадрант III: min(Q), max(K) → обратный контрастный
Квадрант IV:  min(Q), min(K) → слабый positive (фоновый)
```

**Объём:** ~80 строк.

**Техническая значимость:** Существующий **differential attention** (diff_attn.py в проекте!) — частный случай квадрантного. Diff attention вычисляет A₁ - A₂. Квадрантный вычисляет (A₁, A₂, A₃, A₄) с обучаемыми весами.

### 6.3. Лента Мёбиуса как топология attention

**Что:** Беляев показывает, что перекладины вращающегося креста порождают **двойные спирали ленты Мёбиуса**. Цветок Жизни «вышит» ленточками Мёбиуса.

**Для чего в коде:** `MobiusAttentionPattern` — attention-паттерн с Мёбиусовой топологией:
- После полного обхода последовательности (T позиций), attention «переворачивается»
- Токен на позиции T+1 видит токены в «инвертированном» порядке
- Это **неориентируемая поверхность** в пространстве attention

Конкретно: attention mask, где вторая половина последовательности «видит» первую в обратном порядке.

**Объём:** ~60 строк.

### 6.4. Кварковая модель и 8 триграмм

**Что:** Беляев показывает, что мезонные семейства элементарных частиц описываются дуадной декартовой системой ⟨u|d|s⟩ ↔ ⟨ū|d̄|s̄⟩, а барионные — триадной ⟨u|u|d⟩ ↔ ⟨d|s|s⟩. Обе системы — это В ТОЧНОСТИ два типа координат 8 вершин куба (= 8 триграмм).

**Для чего в коде:** Два режима attention-головы:
- **Мезонный** (дуадный): голова сравнивает Q с АНТИПОДОМ K (Q↔K̄)
- **Барионный** (триадный): голова сравнивает Q с СОПРЯЖЁННЫМ K (Q↔K*)

```python
class DualModeHead(nn.Module):
    def __init__(self):
        self.mode = nn.Parameter(torch.tensor(0.5))  # 0=мезонный, 1=барионный
    def forward(self, q, k):
        meson = q @ (-k).T      # антиподальный
        baryon = q @ conjugate(k).T  # сопряжённый
        alpha = torch.sigmoid(self.mode)
        return alpha * meson + (1 - alpha) * baryon
```

**Объём:** ~60 строк.

### 6.5. Гиперкуб как «куб из кубов»

**Что:** Беляев показывает структуру Гиперкуба, где каждая вершина базисного куба сама является кубиком: 8 × 8 = 64 = гексаграммы. Это **рекурсивная** структура.

**Для чего в коде:** `RecursiveCubeAttention` — attention на двух уровнях:
- Уровень 1 (внутри кубика): 8 триграмм взаимодействуют друг с другом
- Уровень 2 (между кубиками): 8 кубиков взаимодействуют через своих «представителей»

Это **иерархический attention** с математически обоснованной двухуровневой структурой. Аналог: Set Transformer (Lee et al., 2019), но с фиксированной кубической топологией.

**Объём:** ~100 строк.

### 6.6. Цветок Жизни — 7 базисных кубиков

**Что:** Цветок Жизни Беляева содержит:
- 2 центральных кубика (= «мировая ось»)
- 6 периферийных кубиков
- 12 лепестков = 12 внешних вершин
- 6 внутренних лепестков = 6 вращающихся вершин центра

Всего: 8 кубиков, 18 уникальных связей.

**Для чего в коде:** `FlowerOfLifeGraph` — граф связей для sparse attention:

```python
# 7 кубиков (1 центральный + 6 периферийных)
# Каждый кубик = 8 вершин
# Связи: кубик-кубик через общие вершины (Мёбиусовы лепестки)
# Итого: 56 вершин с 18 типами связей
```

Применение: **Graph Attention Network** (GAT) на графе Цветка Жизни.

**Объём:** ~120 строк.

### 6.7. Платоновы тела и структурные дефекты

**Что:** Беляев показывает: пара кубов (16 вершин) → икосаэдр (12 вершин) = дефект 4 вершин. Все 5 Платоновых тел — дополнительны друг к другу и связаны инвариантными преобразованиями.

**Для чего в коде:** `StructuralDefect` — при объединении двух кодбуков (2 куба → 16 элементов) происходит «сжатие» в 12 элементов (икосаэдр). Дефект 4 — это **информационное сжатие**. Аналог: bottleneck в autoencoder.

```python
class StructuralDefectLayer(nn.Module):
    """Объединяет два 8-элементных представления в 12-элементное."""
    def forward(self, cube1, cube2):  # (B, 8, d), (B, 8, d)
        combined = torch.cat([cube1, cube2], dim=1)  # (B, 16, d)
        return self.compress(combined)  # (B, 12, d) — "икосаэдр"
```

**Объём:** ~70 строк.

**Техническая значимость:** 16 → 12 = потеря 25% элементов. Это КОНСТРУКТИВНЫЙ пример информационного bottleneck с геометрическим обоснованием. В ML аналог: attention pooling с фиксированным коэффициентом сжатия.

### 6.8. «Ткацкий станок» — 4 уровня иерархии матриц

**Что:** Беляев показывает формирование многомерных матриц через 4 уровня:
- Уровень I: базисный крестик (2 элемента)
- Уровень II: пара крестиков → куб (8 элементов)
- Уровень III: куб из кубов (64 элемента)
- Уровень IV: гиперкуб (4096 элементов)

На каждом уровне — собственные «крестики» (attention-паттерны), вложенные в крестики предыдущего уровня.

**Для чего в коде:** `WeavingLoomArchitecture` — 4-уровневая иерархия слоёв:

```
Слой 1:  2 → 2 (binary gate: инь/ян)
Слой 2:  8 → 8 (триграммный куб)
Слой 3:  64 → 64 (гексаграммный гиперкуб)
Слой 4:  4096 → 4096 (мега-гиперкуб)
```

Каждый слой использует attention-паттерн своего уровня:
- Слой 1: скалярный gate (sigmoid)
- Слой 2: attention 8×8 с кубической геометрией
- Слой 3: attention 64×64 с гиперкубной геометрией (= текущий v50)
- Слой 4: attention 4096×4096 — НЕ полный, а block-sparse через иерархию

**Объём:** ~150 строк (самый крупный новый модуль).

## ПЕРЕКРЁСТНЫЕ СВЯЗИ (кросс-ступени)

Следующие темы возникают на ПЕРЕСЕЧЕНИИ нескольких источников.

### X.1. Герман + Андреев: единая формула T(n) = n(n-1)/2

Два автора независимо используют одну формулу. В коде — одна реализация, два применения:
- Герман: для упаковки циклического поля
- Андреев: для матричного расположения гексаграмм

### X.2. Герман + Фомюк: антиподальность n + (P/2) = P+1

Герман доказывает теорему, Фомюк наблюдает эмпирически. В коде:
- Теорема (документация) ← Герман
- Реализация (регуляризация) ← Фомюк

### X.3. Касаткин + Беляев: ось Z = мировая ось вращения

Касаткин выделяет ось Z в кубе; Беляев называет главную диагональ «мировой осью вращения». В коде:
- 3D embedding ← Касаткин
- Привилегированная ось attention ← Беляев

### X.4. Беляев + Герман: магический квадрат из рычажных весов

Герман конструирует магический квадрат из упаковки; Беляев показывает, что рычажные весы формируют квадратные структуры. В коде:
- Алгоритм построения ← Герман
- Физическое обоснование ← Беляев

### X.5. Склярова + Беляев: «Один порождает Два, Два порождает Три»

Склярова описывает иерархию И-Цзин (монада → дуограмма → триграмма → гексаграмма); Беляев цитирует: «Один порождает ДВА, Два порождает ТРИ, а ТРИ порождает тысячу вещей». В коде:
- 4-уровневая иерархия ← Беляев (ткацкий станок)
- Конкретные уровни ← Склярова (1→2→4→8→64)

### X.6. Фомюк + Беляев: E8 vs гиперкуб (контраст)

Из analysis-habr-1005298.md: E8 вносит <1% в вычисления модели.
Из Беляева: коллизии при P=240.
Из Германа: P=240 не является 2^k.
В совокупности — **тройной аргумент** против E8 и в пользу гиперкуба.
