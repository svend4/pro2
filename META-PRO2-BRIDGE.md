# meta → pro2: Мост между верификацией и обучением

**Дата:** 2026-03-20
**Ветка:** `claude/dev-status-review-oOWWQ`
**Статус:** Концептуальный анализ — готов к имплементации

---

## Суть документа

`meta` — это доказательная база для `pro2`: математика Q6, реализованная чисто, без нейросетевого шума, с нулём внешних зависимостей и полным покрытием тестами (1707 тестов, 27 файлов). Каждая из семи открытых проблем `pro2` имеет соответствующий рабочий модуль в `meta`.

**Логика переноса:**

> Прежде чем встраивать геометрию Q6 в трансформер, нужно убедиться,
> что эта геометрия работает корректно как самостоятельная математическая
> система. meta — это именно такая верификация. Только после того, как
> hexlearn, hexnet, hexphys и hexsym дают воспроизводимые, протестированные
> результаты, их можно переносить в нейросетевую архитектуру pro2 — с
> уверенностью, что проблема находится в слое обучения, а не в самой геометрии.

## Детальный разбор по каждой проблеме

### Проблема 1: Информационный bottleneck — один общий `trit_proj`

**В pro2 (v60–v61):**

Корневая причина того, что Interlingua не работает — один общий проектор `trit_proj` применяется ко всем модулям. Все 64 архетипа получают идентичные значения. Все модули «голосуют» одинаково. Readout attention видит 64 неразличимых вектора → выход Interlingua сводится к константе → `gate` стабилизируется на 0.49.

```
1. Общий trit_proj для всех модулей
   ↓
2. Все модули проецируются через одну матрицу → триты одинаковые
   ↓
3. Consensus = sign(N × one_trit) = тот же trit
   ↓
4. 64 архетипа — все идентичны
   ↓
5. Readout attention видит 64 копии → выход = avg(одинаковых) = константа
   ↓
6. PPL ≈ vanilla
```

**Решение в meta — `hexlearn`:**

k-NN, k-медоиды, наивный Байес и цепи Маркова работают непосредственно на вершинах Q6, где каждая из 64 точек геометрически различима по метрике Хэмминга. Дифференциация возникает не из обученных весов, а из самой структуры пространства.

Если проекция выполняется на геометрически неравные точки, разные входы дадут разные ближайшие вершины **автоматически** — это устраняет возможность коллапса к единому голосованию на уровне геометрии, до обучения.

**Перенос в pro2:** заменить общий `trit_proj` на геометрическую проекцию типа k-NN на Q6. Тесты hexlearn верифицируют, что такая проекция даёт дифференцированные ближайшие вершины для разных входов.

### Проблема 2: Gate ≈ 0.5 — Q6-routing не реализован

**В pro2:**

Теоретический Q6-router (маршрутизация сигналов через гиперкуб по кратчайшим путям Хэмминга) описан в `CONCEPTUAL_STAGE.md`, но ни одного рабочего файла, реализующего этот маршрутизатор, в pro2 не существовало. Вместо этого — gate с инициализацией 0.5, который смещается к 0.49 (слабое предпочтение «не использовать»).

**Решение в meta — `hexnet`:**

Маршрутизация непосредственно на Q6: кратчайшие пути через расстояние Хэмминга, броадкаст, перколяция, симуляция трафика. Это рабочий, протестированный модуль.

**Перенос в pro2:** заменить скалярный `global_gate` на Q6-router из hexnet. Вместо «использовать или нет?» модель выбирает «через какие вершины Q6 маршрутизировать?» — это содержательный выбор, а не бинарный шум.

### Проблема 3: Temperature annealing не доходил до конца

**В pro2:**

К шагу 800 температура снижалась лишь до 0.689 (целевое: 0.1 при warmup_steps=2000). Модель никогда не работала с дискретными тритами. Даже после фикса STE — 64 архетипа активировались, но все триты оказались единодушными (либо все +1, либо все 0, либо все -1).

```
Шаг   800:  T = 0.689  ← эксперименты остановились здесь
Шаг  2000:  T = 0.100  ← дискретная фаза (не достигнута)
```

**Решение в meta — `hexphys`:**

Реализует цепочку Изинга, Монте-Карло (алгоритм Метрополиса) и квантовые корреляторы на Q6. Метрополис-MCMC — корректный математический фундамент для temperature annealing: алгоритм принимает или отвергает переходы между дискретными состояниями с вероятностью exp(−ΔE/T), и делает это аналитически верно.

В pro2 расписание annealing подбиралось эмпирически. В meta физика перехода между дискретными состояниями — отдельный верифицированный модуль с тестами.

**Перенос в pro2:** использовать параметры Метрополис-MCMC из hexphys как основу для расписания TernaryQuantizer. Критически важно: параметры T_init, T_min, schedule_type должны соответствовать реальной физике перехода, а не произвольным константам.

### Проблема 4: CrossDomainAnalogy — 15 из 36 клеток

**В pro2:**

Из полной матрицы 6×6 = 36 клеток (6 источников × 6 источников) CrossDomainAnalogy реализовывала только 15 — C(6,2) без диагонали и без обратных направлений. Само-осведомлённость каждого эксперта (диагональ 6 клеток) отсутствовала полностью. Симметричность (A→B vs B→A) не была учтена.

**Решение в meta — `hexsym`:**

Реализует полную группу автоморфизмов `Aut(Q6) = B₆ = S₆ ⋉ (Z₂)⁶`, орбиты под её действием и лемму Бернсайда для подсчёта попарных эквивалентностей.

Группа автоморфизмов отвечает на вопрос: «какие из 36 клеток матрицы 6×6 эквивалентны по симметрии?» Вместо того чтобы вручную указывать 15 пар, команда:

```bash
python3 projects/hexsym/hexsym.py orbits
```

выдаёт полную, математически обоснованную карту всех взаимодействий — с указанием, какие пары симметричны, какие уникальны, и сколько независимых параметров нужно для полной параметризации.

**Перенос в pro2:** использовать вывод hexsym для построения полной (36-клеточной) CrossDomainAnalogy с правильной симметризацией.

### Проблема 5: STE zero-gradient trap — неполное решение

**В pro2:**

Технический фикс (temperature annealing) снял барьер — 64 архетипа начали активироваться. Но дифференциации так и не возникло: все модули голосовали единодушно. Причина: даже с активными архетипами, если проекция выполнена через общий `trit_proj`, различие между архетипами не возникает.

**Решение в meta — `hexring`:**

Реализует Walsh-Hadamard Transform (WHT), ANF-представление и bent-функции на Q6.

WHT — спектральное преобразование на `{−1,+1}⁶`:
- Дифференцируемо всюду (zero-gradient trap невозможен)
- Работает в пространстве Q6
- Bent-функции (функции с максимально плоским WHT-спектром) — теоретически оптимальные архетипы: каждый из 64 имеет равные веса по всему спектру, что структурно **гарантирует дифференциацию**

**Перенос в pro2:** рассмотреть WHT-проекцию как дифференцируемую альтернативу `TernaryQuantizer` для начальной фазы обучения. Bent-функции использовать как инициализацию архетипного кодбука вместо случайного или Q6-фиксированного.

### Проблема 6: Q4⊂Q6 вложение отсутствовало

**В pro2:**

PseudoRAG (Q4, 16 архетипов) математически вложен в Q6 как подструктура — это доказано теоретически. Но ни одного файла, реализующего это вложение, не существовало. PseudoRAG оставался изолированным прообразом без явной связи с Q6-архитектурой.

**Решение в meta — `hexdim`:**

Явно реализует все вложения измерений: тессеракты Q4, куб Q3, разложение Q6 = Q3 × Q3 и псевдо-QR. Команда:

```bash
python3 projects/hexdim/hexdim.py tesseracts
```

выводит все 15 копий Q4 внутри Q6 — готово, протестировано, с CLI.

**Перенос в pro2:** использовать hexdim для явного задания Q4-подпространств внутри Q6. PseudoRAG-архетипы (16 штук) можно интерпретировать как одно из 15 тессерактных вложений — это даёт им математически обоснованное место в Q6.

---

### Проблема 7: 41 изолированный utils-файл без интеграции

**В pro2:**

Накопилось 41 экспериментальный модуль `utils_v12..v52` без единого dependency graph и без интеграционных тестов между файлами. Каждая новая идея создавала новый файл, не взаимодействующий с предыдущими.

**Решение в meta — `hexcore`:**

Монорепо построено вокруг единой библиотеки `libs/hexcore` без внешних зависимостей. Все 24 проекта импортируют из неё. 1707 тестов в 27 файлах — все проходят. Команда `make test` верифицирует всю экосистему разом.

В pro2 знания накапливаются в изолированных файлах. В meta каждый новый проект опирается на одно и то же верифицированное ядро — добавление нового компонента автоматически покрывается системой тестов.

**Перенос в pro2:** при следующей итерации архитектуры использовать hexcore как образец организации: единая библиотека геометрических примитивов Q6 вместо 41 utils-файла. Это не рефакторинг ради рефакторинга — это предусловие для доверия экспериментальным результатам.

## Ксерокс-тест: система, способная отвечать на вопросы о себе

`hexnav` — интерактивный навигатор по Q6. Команда:

```bash
python3 projects/hexnav/hexnav.py 42
```

показывает гексаграмму 42, её соседей по Хэммингу (расстояние 1–6), кратчайшие пути и семантику. Система может ответить на вопрос «что такое вершина 42, какие у неё соседи, и каков кратчайший путь до вершины 63?» — причём ответ будет верифицированным, а не аппроксимацией нейросети.

Это и есть рабочий ксерокс-тест: Q6-геометрия, которую pro2 использует как `inductive bias`, доступна для прямого инспектирования — не через модель, а через математику.

## Протокол переноса (рекомендуемый порядок)

```
Шаг 1 — hexcore → pro2
  Создать libs/q6core в pro2 по образцу hexcore.
  Перенести: Q6-примитивы, метрику Хэмминга, базовые операции на {-1,+1}^6.
  Добавить тесты. Только после прохождения тестов — переходить к шагу 2.

Шаг 2 — hexlearn → pro2
  Реализовать геометрическую проекцию (k-NN на Q6) вместо общего trit_proj.
  Верифицировать: разные входы → разные ближайшие вершины.
  Ожидаемый эффект: дифференциация тритов, gate смещается от 0.5.

Шаг 3 — hexring (WHT) → pro2
  Добавить WHT-проекцию как дифференцируемую альтернативу TernaryQuantizer.
  Инициализировать архетипный кодбук bent-функциями из hexring.
  Ожидаемый эффект: исчезновение zero-gradient trap структурно, а не через annealing.

Шаг 4 — hexphys → pro2
  Использовать параметры Метрополис-MCMC для обоснованного расписания температуры.
  Ожидаемый эффект: корректный переход в дискретную фазу к шагу ~1200.

Шаг 5 — hexsym → pro2
  Построить полную CrossDomainAnalogy (36 клеток) с орбитами Aut(Q6).
  Ожидаемый эффект: правильная параметризация взаимодействий всех источников.

Шаг 6 — hexnet → pro2
  Заменить scalar gate на Q6-router.
  Ожидаемый эффект: содержательная маршрутизация вместо бинарного переключателя.
```
